# 0308. N9 multi-target CI — arm64 lane (N9-A/B)

- **Status:** Accepted (workflows landed; image bake + hard-fail flip pending CI)
- **Kind:** Architecture Decision
- **Date:** 2026-07-09
- **Deciders:** ShortArrow

## Plan revision vs roadmap 2026-06-27/07-03

The roadmap's N9-A was "aarch64-linux cross-compile from x86_64." Superseded: GitHub's free public-repo **native arm64 runners** (`ubuntu-24.04-arm`) make cross-compilation unnecessary for CI validation — `llc` targets the host triple, so a native arm64 test lane exercises the entire compile-link-run pipeline (every e2e test builds and executes an aarch64 binary). Cross-compile support in `link.rs` stays out of scope until a concrete artifact need arises (N9-D).

## The image problem

The existing CI image (ADR 0132) is `archlinux:base-devel` + AUR mlir — and Arch has **no official arm64 container**. Options considered:

- Arch Linux ARM community images — provenance too weak for the ADR 0126 supply-chain posture. Rejected.
- MLIR source build on Ubuntu arm64 — works but ~1-2h bake per revision. Fallback if apt packages disappoint.
- **apt.llvm.org LLVM/MLIR 22 packages on Ubuntu 24.04 arm64 — chosen.** `libmlir-22-dev` + `mlir-22-tools` + `llvm-22-dev` ship arm64 builds; image bakes in minutes.

Consequence: the two lanes intentionally differ in distro (Arch/amd64 vs Ubuntu/arm64). That divergence is a feature — it also covers the "works outside Arch" portability question ADR 0132 deferred.

## Landed in this ADR

- `.github/docker/Dockerfile.ci-arm64` — Ubuntu 24.04 + apt.llvm.org LLVM/MLIR 22 + rustup; `MLIR_SYS_220_PREFIX=/usr/lib/llvm-22`.
- `build-ci-image.yml` → two jobs: existing amd64 (unchanged tags) + `build-arm64` pushing `ghcr.io/shortarrow/lumelir-ci:mlir-22-arm64`, built natively on `ubuntu-24.04-arm` (no qemu).
- `ci.yml` → new `clippy-and-test-arm64` job on `ubuntu-24.04-arm` in the arm64 image. **`continue-on-error: true` during bootstrap** — the image doesn't exist until the image workflow's first run completes.

## Follow-up (bake sequence)

1. This push triggers `build-ci-image` (path filter matches); arm64 image bakes.
2. Next main push (or re-run) exercises `clippy-and-test-arm64` for real.
3. When green: flip `continue-on-error` off — that flip commit closes **N9-B**.
4. **N9-C** (arm64-darwin): `macos-15` runners are Apple Silicon; MLIR via Homebrew LLVM needs a libMLIR-C probe first. Separate ADR.
5. **N9-D** (release artifacts): tag-triggered job producing lumelir binaries per target. Separate ADR.

## Bake log (2026-07-09)

1. `rustup-init --component clippy rustfmt` → space-separated components rejected; fixed to comma form (`1af6481`).
2. `tblgen` build: missing static `Polly` → apt.llvm.org splits it into `libpolly-22-dev`; added (`0cd1c07`).
3. Image bakes green in ~6 min (vs the amd64 lane's AUR-source hours). Lane compiles, clippy passes, full suite runs.
4. **First genuine multi-arch bug caught**: `tests/phase2_5c2_kind_captures.rs` — 3 tests (`bool_upvalue_*`, `nil_upvalue_*`) exit non-zero on aarch64 only. Suspect: i1 (Bool/Nil) upvalue values crossing the closure ABI — x86-64 and AArch64 differ on who extends sub-register integers (AAPCS64 leaves high bits unspecified; SysV amd64 callers commonly zero-extend). All other ~150 suites green on arm64.

The lane stays `continue-on-error: true` until the i1 fix lands (**N9-E**, needs codegen work in the upvalue box store/load path — explicit `zext`/`trunc` at the boundary). The flip commit then closes N9-B.

## Risks

| Risk | Mitigation |
|---|---|
| apt.llvm.org `libmlir-22-dev` lacks `libMLIR-C.so` | Image build fails loudly → fall back to `-DMLIR_BUILD_MLIR_C_DYLIB=ON` source build stage (the amd64 AUR patch precedent) |
| melior 0.27 expects different prefix layout on Debian-style multi-arch | `MLIR_SYS_220_PREFIX=/usr/lib/llvm-22` matches apt.llvm.org's install root; adjust from CI logs if mlir-sys probing fails |
| arm64 runner queue times | Public-repo arm runners are GA; acceptable for a non-blocking lane |
